# R线策略中台 V2

面向策略、产品、数据、教研、学服和二销联合评审的静态后台原型。它把用户行为进入、计分、H 分层、F12 准入、角色路由、任务回写、次日重算和效果复盘串成一条可下钻链路，用来说明公司版系统应如何在现有后台能力上落地。

在线原型：<https://viannni.github.io/rline-strategy-center-pages/>

规则事实源：[R线用户分数系统与高优用户识别方案（最新版）](https://alidocs.dingtalk.com/i/nodes/dQPGYqjpJYgoM27OcBp667plWakx1Z5N?utm_scene=person_space)

## 使用边界

- 页面中的用户、任务、分数和回写均为脱敏模拟数据，不连接公司生产数据。
- “演示视图”只切换策略、Agent、学情和二销的信息范围，不代表真实权限控制。
- 原型展示产品逻辑和字段契约，不替代产研对接口、字段口径、权限及刷新时效的最终确认。
- 请勿向本地导入真实姓名、手机号、订单号、沟通记录或其他个人信息。

## 设计原则

首版优先复用营销中心、CRM 标签、现有任务入口、教务中心、工单、外呼、业务卡片和数据看板。证据状态在“系统落位”中明确区分：

- **已确认可复用**：已有资料明确证明入口或能力存在。
- **已有入口待核对**：已看到系统入口，但深层字段、接口或权限尚未实读确认。
- **需要改造**：可复用现有载体，但需要增加 R 线字段或用户级回写。
- **必须新增**：跨系统规则计算、快照、准入和路由编排等现状缺口。
- **可降级**：MVP 可先以离线名单、标签交集、固定枚举或定时导入承接。

任何“已有入口待核对”都不能被当作生产能力已经上线。

## 评分与路由

基础分由学习健康 `30`、课程体验 `15`、成果外化 `15`、家长互动 `15`、用户适配 `5` 五个维度按可用分母标准化为 `0-100`。系统同时保留逐项原始得分、维度得分和标准化后的 `rawBaseScore`；F15 风险扣分后形成最终 `baseScore`，单次总扣分上限为 `30`。

F13 营销意向和 F14 交易状态是独立信号，不进入基础分；F12 是触达准入和频控闸门；F15 风险熔断优先于分数、营销意向、交易优先级及销售任务。H4 先判定风险修复，其后依次判断 H1、H2、H3 和 L。

F16 回写分两段生效：营销意向、交易状态和风险变化实时更新；家长互动等基础分贡献进入待应用记录，在“模拟次日重算”后才写入基础分。这样可避免长期行为分与即时转化信号混算。

## 演示路径

1. 在“总控台”查看行为进入到任务回写的全链路和各角色待办。
2. 在“用户中心”打开任一用户，核对逐项计分证据、F13/F14、F12、风险、H 层级、主责团队和当前任务。
3. 切换“二销承接”视图，在“角色任务台”提交一次结构化 F16 回写，再查看次日重算前后差异。
4. 在“进线中心”核对文字/电话渠道、绑定模式、主责小组、SLA 和七步路由轨迹。
5. 在“评分中心”逐条核对指标分值、取数窗口、字段来源、缺失处理和规则版本。
6. 在“系统落位”和“提需清单”按现有能力、改造项、缺口、降级方案和验收口径与产研逐项讨论。

## 本地运行

要求 Node.js 20+ 与 Python 3：

```bash
npm test
npm run serve
```

打开 <http://127.0.0.1:4173/>。页面使用版本化 `localStorage` 保存模拟操作；可在顶部撤销最近一次改动或重置演示数据。浏览器存储不可用时会退化为内存模式。

## 目录说明

- `core/`：计分、路由、导入导出和状态存储。
- `data/`：规则、模拟用户和现有系统能力证据。
- `views/`：11 个业务视图及其交互。
- `tests/`：规则、角色边界、回写、导入、安全与 UI 契约测试。
- `output/playwright/`：桌面端与移动端验收截图。

GitHub Pages 从仓库 `main` 分支根目录发布，无服务端依赖。
