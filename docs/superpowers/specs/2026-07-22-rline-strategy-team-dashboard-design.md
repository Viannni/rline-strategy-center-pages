# R线策略团队经营后台设计规格

## 1. 文档状态

- 状态：已按用户最新方向重写，待用户复核后进入实施计划。
- 日期：2026-07-22。
- 目标站点：[R线策略中台后台](https://viannni.github.io/rline-strategy-center-pages/)。
- 本规格取代当前迭代中的“进线中心、指导师作业台、二销作业台、48期数据验证优先”的设计方向。
- 当前迭代暂停K2 48期数据导入模拟；后续如需验证评分系统，再作为独立验证模块接回。

## 2. 新定位

本后台不是一线服务系统，也不是销售或指导师工作台，而是策略团队用于管理R线全周期策略的经营中台。

策略团队在这里完成五件事：

1. 看清楚R线全周期是否有策略覆盖，哪里过密、哪里空白。
2. 配置可执行的策略资产，包括内容、活动、AI场景、触达规则、模型规则、用户分层规则。
3. 圈选策略人群，并把策略包下发给后续系统执行。
4. 追踪策略是否被执行、执行是否完整、数据是否回写。
5. 用回流数据判断策略是否有效，并反向校准内容、模型、触达节奏和产研需求。

核心链路：

```text
策略目标
-> 策略资产配置
-> 人群圈选与准入判断
-> 策略包下发
-> 执行状态回流
-> 用户行为/触达/交易/风险数据回写
-> 策略有效性评估
-> 策略迭代与产研提需
```

## 3. 本期明确不做

以下能力不在本次后台范围中，即使已有旧页面也要降级或移除：

- 不做进线中心派单页面。
- 不做Agent、指导师、学情老师、二销老师的一线作业台。
- 不做个人老师待办列表。
- 不做用户与老师/IP绑定管理。
- 不做真实私聊发送、外呼、工单创建、CRM任务创建。
- 不展示真实姓名、私聊原文、电话录音、IP归属或老师归属。
- 不把K2 48期真实明细发布到GitHub Pages。

这些能力由产研在独立执行系统中建设；本后台只查看策略层面的下发、执行、回写和复盘结果。

## 4. 角色与工作区

后台按“策略角色产出”组织，而不是按用户生命周期或一线团队组织。

| 策略角色 | 核心产出 | 后台工作区 |
|---|---|---|
| 内容策略 | 活动、讲座、复习直播、月测、PK、成长报告、奖学金内容资产 | 内容策略工作区 |
| 应用策略 | AI场景落地、Agent知识库、自动化服务能力 | 应用策略工作区 |
| 执行策略 | 非绑定用户中心化触达策略、触达节奏、SOP动作包 | 执行策略工作区 |
| 模型策略 | 续费模型、绑定窗口策略、关单SOP、阈值校准 | 模型策略工作区 |
| 用户生命周期洞察 | 积分/奖学金、用户画像、行为归因、信号识别、评分规则 | 用户洞察工作区 |

用户本人主要负责“模型策略 + 用户生命周期洞察”，但后台必须能看见其他角色的配置和进度，因为模型与洞察要依赖内容、执行、AI应用的回流数据。

| 一级模块 | 解决的问题 | 主要使用者 |

| 策略资产库 | 所有策略、内容、SOP、模型、规则是否可查、可复用、可版本化 | 全策略团队 |
| 内容策略 | 活动/内容资产如何配置、排期、复盘 | 内容策略 |
| 应用策略 | AI场景覆盖、解决率、知识库缺口、人工兜底问题 | 应用策略 |
| 执行策略 | 中心化触达规则、频控、渠道、动作包 | 执行策略 |
| 模型策略 | 续费高优识别、蜜续/结续策略、关单SOP、模型校准 | 模型策略 |

| 人群圈选 | 按生命周期、分数、标签、风险、行为圈策略目标人群 | 全策略团队 |

| 数据底座/产研提需 | 字段、事件、埋点、刷新、接口、权限、验收 | 策略、产研、数据 |

## 6. 策略总控台

策略总控台是领导和策略负责人进入后台后的第一屏，必须展示经营问题，而不是堆字段。

核心指标：

| 指标 | 含义 |
|---|---|
| 生命周期覆盖率 | T0-T28、M1-M12每个节点是否有内容/触达/模型/复盘策略 |
| 策略资产上线率 | 已上线策略数 / 已确认策略数 |
| 执行完整率 | 已完成触达、活动、AI场景、二销承接等执行量 / 应执行量 |
| 回写完整率 | 执行结果、用户行为、风险、交易等回写字段完整度 |
| 高优用户识别稳定度 | H1/H2用户命中、迁移、续费/未续对照情况 |
| 触达冲突数 | 同用户同周期多策略冲突、频控超限、风险未排除 |
| 策略有效策略占比 | 达到观察窗口后指标正向的策略占比 |
| 数据异常数 | 延迟、缺失、口径不一致、事件未入库 |

第一屏下钻：

- 生命周期热力图：按月课T0-T28、年课M1-M12展示策略密度。
- 角色产出进度：内容、应用、执行、模型、洞察五类产出状态。
- 本周风险项：字段缺失、策略未执行、效果异常、产研阻塞。
- 重点人群池：高优、潜力、风险、沉默、强意向、领券未付、报告已开未沟通。

## 7. 策略资产库

策略资产库是所有策略的统一目录，避免内容、执行、模型各自用表管理，后续无法追踪。

每个资产至少包含：

| 字段 | 说明 |
|---|---|
| 策略ID | 统一编号，例如 RS-CONT-001、RS-EXEC-003 |
| 策略名称 | 面向业务可读的R线名称 |
| 策略类型 | 内容、活动、AI、触达、模型、评分、奖学金、关单SOP |
| 所属角色 | 内容策略、应用策略、执行策略、模型策略、用户洞察 |
| 适用产品 | 月课、年课、通用 |
| 生命周期节点 | T0-T28、M1-M12或全周期 |
| 目标人群 | 圈选条件摘要 |
| 排除条件 | 风险、投诉、退款、频控、已购等 |
| 动作内容 | 发送卡片、活动邀请、报告引导、讲座、电话准入、AI服务等 |
| 观察窗口 | 实时、T+1、近7日、节点后、续费窗口内 |
| 核心指标 | 完课、打开、回复、参与、进线、转化、风险改善等 |
| 数据依赖 | 所需字段、事件、接口、刷新周期 |
| 版本状态 | 草稿、待评审、模拟中、已上线、已停用 |
| 负责人 | 策略Owner |

必须支持按策略ID下钻到：

- 规则明细。
- 命中人群样例。
- 下发批次。
- 执行状态。
- 效果表现。
- 问题记录。
- 产研依赖。

## 8. 五类策略工作区

### 8.1 内容策略工作区

目标：管理能够提升学习健康、效果外化和续费认知的内容资产。

页面能力：

- 内容日历：活动、节日比赛、单词PK、复习直播、讲座、月测、成长报告排期。
- 内容资产库：每个内容资产绑定适用节点、人群、触发条件、权益/奖学金规则。
- 内容策略配置：配置“什么人、什么节点、看见什么内容、用什么入口进入”。
- 内容效果复盘：报名、参与、完成、报告打开、学习行为提升、进线、续费信号变化。
- 内容缺口：某节点没有可用内容、某类人群没有合适内容、某内容无回写字段。

重点不是展示漂亮活动页，而是让策略知道：这项内容是否真的改变了用户行为。

### 8.2 应用策略工作区

目标：管理AI和Agent在R线服务中的可覆盖场景，降低人工消耗并提升标准问题解决率。

页面能力：

- AI场景地图：课程入口、活动规则、报告解读、奖学金说明、常规售后、学习提醒。
- Agent知识库命中：问题类型、命中率、解决率、转人工率。
- 兜底问题池：AI无法回答、用户反复追问、转人工后仍未解决的问题。
- AI动作策略：哪些人群先用AI解释，哪些场景必须人工介入。
- 产研/知识库提需：新增意图、模板、卡片、字段、埋点。

应用策略不负责派单，只负责定义AI可处理边界、策略话术资产和异常反馈。

### 8.3 执行策略工作区

目标：管理非绑定用户的中心化触达策略，不直接管理一线老师任务。

页面能力：

- 触达规则库：节点、人群、渠道、文案版本、卡片、频控、排除条件。
- 周期触达日历：同一人群在一周内会收到哪些策略，是否冲突。
- 策略动作包：每个动作包包含触发条件、目标、内容、回写、观察窗口。
- 频控与冲突检查：同用户多策略并发、风险用户误触达、续费窗口与学习提醒冲突。
- 执行反馈看板：已提交、已发送、已触达、已回复、失败、未回写。

执行策略只定义“该怎么触达”，不显示具体老师今天要联系谁。

### 8.4 模型策略工作区

目标：管理高优续费识别、蜜续/结续窗口、绑定二销策略、关单SOP和模型校准。

页面能力：

- 高优识别配置：H1/H2/H3/H4阈值、加分项、扣分项、风险熔断。
- 续费窗口管理：月课T22-T28，年课M8-M12，区分蜜续与结续。
- 绑定准入规则：什么条件下从非绑定中心化触达进入二销绑定承接。
- 关单SOP：高优未续、领券未付、报告已看未转化、强意向未支付、风险修复后回流。
- 模型校准：高分未续、低分续费、误判原因、阈值漂移、字段贡献解释。
- 二销策略反馈：仅展示绑定窗口的策略效果，不做二销个人工作台。

模型策略必须能回答：为什么这个人应该被识别为高优，为什么现在该转二销，为什么没续。

### 8.5 用户洞察工作区

目标：管理用户全生命周期画像、积分/奖学金、行为归因和信号识别。

页面能力：

- 用户画像：学习健康、成果外化、家长互动、活动参与、转化信号、风险状态。
- 分数规则：简化版用户分数、高优分、提分潜力、标签贡献说明。
- 奖学金规则：月课/年课获得、展示、提醒、兑换、使用上限、退还逻辑。
- 信号识别：首课、连续学习、完课率、月测、报告打开、活动参与、回复、领券、支付异议。
- 行为归因：某个策略动作后，用户分数、标签、行为是否变化。
- 人群迁移：H层级、活跃状态、风险状态、续费意向的迁移路径。

用户洞察不只是看用户明细，而是要解释“哪些行为真的代表用户价值和续费可能”。

## 9. 人群圈选

人群圈选是策略团队生成策略包的地方，不是客服派单列表。

必须支持以下条件块：

| 条件块 | 示例 |
|---|---|
| 基础信息 | 产品类型、班期、生命周期节点、购课类型 |
| 学习健康 | 首课、完课率、连续学习、漏学、活跃天数 |
| 效果外化 | 月测、阶段测、成长报告、词汇/绘本/句型累计 |
| 内容参与 | 活动报名、活动完成、讲座到场、复习直播参与 |
| 家长互动 | 打开、点击、回复、预约、拒绝、沉默 |
| 奖学金 | 已获得、接近上限、可兑换、已提示未用 |
| 转化信号 | 领券、询价、路径咨询、支付页访问、未支付 |
| 风险排除 | 退款、投诉、售后未完结、强负向、频控超限 |
| 模型结果 | 高优分、H层级、提分潜力、置信度、标签贡献 |

圈选结果需要展示：

- 命中人数。
- 排除人数及原因。
- 与其他策略人群的重叠率。
- 数据新鲜度。
- 样例用户的匿名解释。
- 可下发动作。
- 预计观察窗口。

## 10. 下发追踪

下发追踪只关注策略包状态，不关注一线老师个人工作量。

每个下发批次字段：

| 字段 | 说明 |
|---|---|
| 批次ID | 每次下发的唯一编号 |
| 策略ID/版本 | 对应策略资产 |
| 人群包ID | 对应圈选结果 |
| 下发系统 | 活动系统、私聊系统、AI系统、二销系统、数据系统 |
| 下发状态 | 草稿、待审批、已提交、系统接收、执行中、完成、失败 |
| 计划人数 | 圈选后可触达人群 |
| 实际触达人数 | 实际发送/展示/进入的人数 |
| 失败原因 | 字段缺失、接口失败、频控、用户不可达、风险拦截 |
| 回写状态 | 未回写、部分回写、已回写、回写异常 |
| 观察窗口 | 该批次何时进入效果复盘 |

下发后必须能回到策略资产库，形成“策略版本 -> 下发批次 -> 效果结果”的闭环。

## 11. 有效性看板

有效性看板参考K2现有中心化SOP思路，但改成R线策略团队视角。

四个核心视图：

1. 策略覆盖视图：按项目、科目、阶段、时间、策略类型查看覆盖密度和空白点。
2. 触达周期视图：按用户、节点、渠道、策略组查看触达频次、冲突和疲劳风险。
3. SOP效果视图：按策略ID、版本、人群、渠道、节点查看打开、点击、回复、参与、完课、报告打开、进线、转化。
4. 实验与校准视图：对比不同版本策略的效果，记录阈值和文案是否需要调整。

所有效果必须绑定观察窗口：

| 类型 | 观察窗口 |
|---|---|
| AI/服务解答 | 2小时、24小时 |
| 私聊触达 | 24小时、3天 |
| 活动/讲座 | 活动结束后1天、7天 |
| 学习行为 | 近7天、近14天 |
| 续费转化 | 续费窗口内、窗口结束后 |
| 风险修复 | 7天、14天 |

## 12. 进线复盘

进线复盘只用于策略反馈，不做进线派单。

要回答的问题：

- 用户为什么进线：活动、报告、奖学金、学习问题、售后问题、续费问题、AI未解决。
- 哪个策略导致进线：策略ID、内容ID、触达批次、AI场景。
- 进线质量如何：有效咨询、低价值咨询、风险、投诉、支付意向、学习规划。
- 是否被解决：一次解决、转人工、二次进线、仍未解决。
- 对策略的影响：保留、改文案、改入口、改频控、改AI知识库、改售后流程。

进线复盘字段：

| 字段 | 说明 |
|---|---|
| 进线来源策略ID | 可为空，但必须尽量归因 |
| 进线类型 | 学习、活动、报告、奖学金、售后、续费、AI兜底 |
| 质量评级 | 高价值、普通、低价值、风险 |
| 是否解决 | 已解决、部分解决、未解决 |
| 是否影响分数 | 加分、扣分、不影响、需人工复核 |
| 策略修正建议 | 文案、入口、频控、内容、模型、产研 |

## 13. 数据底座与产研提需

数据底座要让产研一眼看清楚“已有能力、缺口、刷新周期、验收口径”。

关键数据能力：

| 能力 | 说明 |
|---|---|
| 事件字典 | 课程、活动、报告、AI、触达、交易、风险事件统一命名 |
| 跨系统用户ID | 能把课程、触达、活动、交易、售后数据串起来 |
| 生命周期节点 | 月课T0-T28、年课M1-M12自动识别 |
| 分数快照 | 用户分数、H层级、标签贡献、历史变动 |
| 策略ID/版本ID | 每个触达、活动、AI动作都能追溯到策略版本 |
| 人群包ID | 每次圈选都有可复盘的固定人群结果 |
| 下发批次ID | 每次执行都有状态和结果 |
| 触达回写 | 发送、到达、打开、点击、回复、失败、频控 |
| 活动回写 | 报名、到场、完成、获奖、奖励发放 |
| 报告回写 | 生成、打开、分享、点击下一步 |
| AI回写 | 命中意图、是否解决、是否转人工、失败原因 |
| 交易回写 | 领券、支付页、支付、未支付、退款 |
| 风险回写 | 投诉、售后未完结、退款、强负向、解除风险 |

产研提需清单要支持：

- 当前是否已有。
- 当前后台编号/名称/路径。
- 字段类型。
- 刷新周期。
- 取数口径。
- 是否可替代。
- 替代方案可信度。
- 新增原因。
- 优先级。
- 验收标准。
- 依赖部门。

## 14. 旧版页面改造原则

| 旧页面/能力 | 新处理方式 |
|---|---|
| 进线中心 | 改为进线复盘，只看策略归因和质量 |
| 任务台 | 改为下发追踪，只看策略包执行状态 |
| 用户中心 | 改为人群圈选和匿名样例解释 |
| 生命周期 | 改为策略覆盖地图，服务于策略密度与节点缺口 |
| 评分中心 | 合入用户洞察，保留规则、贡献、迁移、校准 |
| 提分运营 | 拆到内容策略、执行策略、用户洞察 |
| 系统地图 | 改为数据底座/产研提需 |
| Agent/学情/二销角色视图 | 移除一线作业属性，仅在策略归因中作为下发系统或承接类型出现 |

## 15. 视觉与交互方向

这是策略经营后台，视觉应更像“运营驾驶舱 + 策略配置台”，而不是客服系统。

设计要求：

- 第一屏密度高、可扫描，少用营销式大卡片。
- 页面要有可下钻结构：总览 -> 模块 -> 策略ID -> 批次/人群/效果。
- 所有策略配置都要展示“目标、规则、排除、动作、回写、观察窗口、负责人”。
- 所有指标都要显示刷新周期和数据状态。
- 对产研有用的字段要能直接复制成需求表。
- 模拟数据必须匿名，不展示真实身份。

## 16. 成功标准

本轮完成后，用户可以把网站发给产研和项目组，表达清楚：

- R线策略团队后台应该管什么，不应该管什么。
- 五类策略角色分别在后台看什么、配什么、复盘什么。
- 策略动作从配置到圈人、下发、执行、回写、复盘的完整闭环。
- 哪些页面是策略团队真正需要的核心面板。
- 哪些字段和系统能力是产研必须补齐的。
- 进线中心和一线工作台为什么不属于本后台范围。
